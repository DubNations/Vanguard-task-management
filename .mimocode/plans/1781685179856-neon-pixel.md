## 三、分模块详细审计报告

### 3.1 架构层

**优点**:
- 清晰的分层: models → serializers → views → services
- 自定义 User 模型 + UUID 主键
- 合理的 settings 拆分 (base/dev/prod)

**问题**:
- 权限过滤逻辑散落在 6+ 视图中，未抽公共 mixin
- `sys.path.insert(0, APPS_DIR)` 脆弱的路径注入
- 前端 API 模块层 (`api/modules.ts`) 几乎完全未使用，视图直接调用 `api.get/post`
- 前端 `any` 类型泛滥 (27+ 处)
- 后端 `except Exception: pass` 吞掉异常 (15+ 处)
- 前端空 catch 块 (15+ 处)

### 3.2 安全层

**问题清单**:
- 文件上传仅检查扩展名黑名单，`python-magic` 在依赖中但未使用
- 慢查询日志可能记录包含用户数据的 SQL
- 备份文件明文存储无加密
- 缺少 CSP、HSTS、Permissions-Policy 安全头
- `window.open()` 打开未验证的外部 URL
- `TaskDetailView` 的 PATCH 请求可绕过状态机直接修改 status
- `seed_data.py` 在 stdout 明文输出密码
- `dev.py` 使用 MD5 密码哈希器（若误用于生产则全线弱哈希）

### 3.3 性能层

**问题清单**:
- `dashboard/get_summary` 执行 8 次独立 COUNT 查询
- `get_weekly_trend` 执行 2N 次查询 (N=周数)
- 导出/导入同步阻塞请求线程
- `_parse_xlsx` 将整个文件加载到内存
- `send_daily_digest_emails` 产生 N+1 查询 (每用户 3-4 次)
- `KanbanView` 缺少 `comments_count`/`files_count` 注解
- Element Plus 全量导入 + 全部图标全局注册 (~500KB+ gzip)
- 未使用的 echarts/vue-echarts/dayjs 依赖 (~850KB)
- 通知列表/导入导出历史无分页

### 3.4 业务功能层

**问题清单**:
- `AuditLog` 模型定义了但中间件只写日志文件不写数据库
- `TaskClaimView` 领取后未触发积分奖励和邮件通知
- `PointService.deduct` 并发下余额可能为负
- `export_generator.py` 迭代 queryset 后再次 count
- `import_executor` 无事务包裹，中途失败数据不一致
- `TaskCommentCreateSerializer` 评论无最大长度限制
- `TaskParticipantDetailView.put` 使用不存在的 `updated_at` 字段
- 灾难演练脚本查询错误的表名 (`tasks` → `tasks_task`, `users` → `accounts_user`)

### 3.5 工程规范层

**问题清单**:
- 前端无测试框架 (vitest/jest 均未配置)
- CI 缺少安全扫描 (safety/bandit)
- CI 前端缺少 lint 和 type-check 步骤
- 缺少 `.dockerignore` 文件
- 硬编码 URL `/api/v1/files/download/{id}/` 和 `/api/v1/exports/{id}/download/`
- 魔法常量: 看板限制 200 条、通知限制 50 条、导出限制 20 条
- `tsconfig.json` 关闭 `noUnusedLocals`/`noUnusedParameters`
- 前后端重复定义状态/优先级常量

### 3.6 运维可观测层

**问题清单**:
- `AuditLogMiddleware` 写日志文件但不写数据库，审计视图返回空
- Gunicorn 错误日志无结构化格式
- 无 Sentry 前端集成 (仅后端 prod.txt 包含 sentry-sdk)
- 备份无校验和 (checksum)
- 更新脚本无回滚机制
- 健康检查脚本缺少 `set -e`

### 3.7 并发与分布式

**问题清单**:
- `generate_task_no` 竞态条件可产生重复编号
- `download_count += 1` 经典读-改-写竞态
- `PointService.deduct/award` balance 读写无锁
- 前端 `AuditLogView` 两个 watcher 可触发并发请求竞态
- 前端通知乐观更新无回滚机制
- 无限流/熔断配置

### 3.8 部署与合规

**问题清单**:
- Nginx static 路径配置与实际 volume 映射不匹配
- `init.sh` 步骤编号不一致 (1/6 → 6/7)
- `disaster-drill.sh` 查询 Django 表名错误
- `deploy/.env.dev` 未被 `.gitignore` 忽略
- PostgreSQL 未启用 SSL
- 容器无资源限制 (CPU/内存)
- Nginx 缺少请求超时配置 (Slowloris 风险)
- 缺少 SECURITY.md 安全策略文件

## 四、统一重构优化方案

### 4.1 后端架构重构

1. **抽取数据隔离 Mixin** — 统一 `is_superuser / role in (LEADER, ADMIN)` 过滤逻辑
2. **统一异常处理** — 将所有 `except Exception: pass` 替换为 `logger.exception()`
3. **审计日志中间件重写** — 同时写入文件日志和数据库 AuditLog 模型
4. **导出/导入异步化** — 使用 django-q2 将生成任务放入后台队列
5. **Dashboard 查询优化** — 使用 `aggregate()` 替代多次 `count()`

### 4.2 前端架构重构

1. **修复 FC-01** — `v="task"` → `v-if="task"` (1 行修改)
2. **按需导入 Element Plus** — 利用已配置的 auto-import 插件，移除全量导入
3. **移除死依赖** — echarts, vue-echarts, dayjs
4. **统一 API 调用层** — 所有视图改用 `api/modules.ts` 中定义的函数
5. **添加错误状态 UI** — 15+ 空 catch 块改为显示错误信息+重试按钮

### 4.3 部署架构优化

1. **容器非 root 运行** — Dockerfile 添加 USER 指令
2. **启用 HTTPS** — certbot + HSTS + 强制重定向
3. **安全头补全** — CSP + Permissions-Policy + server_tokens off
4. **CI/CD 补全** — 安全扫描 + 前端 lint + 依赖缓存
5. **备份加密** — 使用 openssl 或 restic 加密备份文件

## 六、配套落地建议

### 6.1 新增工具

| 工具 | 用途 |
|------|------|
| `bandit` | Python 安全扫描 |
| `safety` | 依赖漏洞检查 |
| `npm audit` | 前端依赖漏洞 |
| `vitest` | 前端单元测试 |
| `pre-commit` | Git 提交前自动检查 |

### 6.2 CI/CD 规则

```yaml
# 建议添加到 .github/workflows/ci.yml
- name: Backend security scan
  run: |
    pip install safety bandit
    safety check -r backend/requirements/base.txt
    bandit -r backend/apps/ -ll

- name: Frontend lint & typecheck
  run: |
    cd frontend
    npm run lint
    npx vue-tsc --noEmit

- name: Frontend audit
  run: |
    cd frontend
    npm audit --production
```

### 6.3 建议新增的 Makefile 目标

```makefile
security:
	cd backend && python -m bandit -r apps/ -ll
	cd backend && python -m safety check -r requirements/base.txt
	cd frontend && npm audit --production

typecheck:
	cd frontend && npx vue-tsc --noEmit
```
